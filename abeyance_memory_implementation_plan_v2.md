## Step 2 — Core System Decomposition

The implementation breaks purely down into 1 database engine, 1 math engine, and an orchestration service mesh.

### Core Engines & Logic Flow
1. **Enrichment Pipeline (`EC`)**: Driven directly from streaming evidence via Kafka/ITSM APIs. Implements the 4-step logic to convert `RawEvidence` to `AbeyanceFragmentORM` (LLD §6).
2. **Fragment Engine (`FS`)**: PG/pgvector persistence layer. Implements fast queries utilizing `GIN(failure_mode_tags)` and `ivfflat(enriched_embedding vector_cosine_ops)` (LLD §14 Databases).
3. **Snap Engine (`SE`)**: Acts instantly upon new fragment insertion. Fetches targeted fragments via SQL bounding logic (LLD §9 SQL), runs the complex weighted formula for the 5 distinct Dark Divergence profiles, and emits `SNAP` (0.75+), `NEAR_MISS` (0.55+), or `AFFINITY_EDGE` (0.40+).
4. **Accumulation Graph Engine (`AG`)**: Detects high-yield weak signals using Recursive Common Table Expressions (CTE). Performs `HAVING count(*) >= 3` grouping before merging sub-graphs through Noisy-OR probabilistic fusion (LLD §10 SQL & Algorithm).
5. **Decay Engine (`DE`)**: Asynchronous worker applying `exp(-age_days / τ)` to everything daily (LLD §11 Decay Formula).
6. **Retrieval Engine**: Hierarchical storage driver routing queries to Parquet if timestamp > 180 days (LLD §12).

## Step 4 — Multi-Agent Implementation System

A fleet of functional agents will break down this complex system into parallel streams.

**Agent 1: Data Model Agent**
- **Responsibilities**: Implement PostgreSQL Data models (LLD §14: `AbeyanceFragmentORM`, `ShadowEntityORM`, `AccumulationEdgeORM`). Design Alembic migrations perfectly applying specific indexing (GIN, ivfflat, jsonb_path_ops).
- **Core LLD Refs**: §5 schema, §14 DB indexes.
- **Output**: Output exactly the SQLAlchemy code from LLD §14.

**Agent 2: Graph Algorithm Agent**
- **Responsibilities**: Build the intricate PostgreSQL logic for `ShadowTopologyService` (2-hop network walk) and `AccumulationGraphService` (Cluster detection).
- **Core LLD Refs**: §8 (2 hops), §10 (recursive CTE example query).
- **Output**: Python repositories dispatching complex SQLAlchemy/Raw-SQL executing the graph walks exactly as specified in §10.

**Agent 3: Embeddings & Math Agent**
- **Responsibilities**: Transliterate the mathematical encoding formulas from strings to functional Python matrix code.
- **Core LLD Refs**: §7 Temporal Encoding (`[sin(2π*h/24), cos(2π*h/24)]`), §11 decay formulas.
- **Output**: `core/math_utils.py` handling exact gaussian `exp(-dist)` calculations and sine/cosine wave array concatenations for 256-dim components.

**Agent 4: Snapping & Fusion Agent**
- **Responsibilities**: Translate pairwise evaluation algorithms mapping dynamic weights against vectors. Also integrate Noisy-OR cluster logic. 
- **Core LLD Refs**: §9 (Scoring formula `w_sem*cos_sim + ... * temp_weight`, weighting matrix), §10 (Noisy-OR equation).
- **Output**: Iterative logic `SnapEngine.evaluate` applying semantic L2 mapping and Jaccard intersection math.

**Agent 5: API & Orchestration Agent**
- **Responsibilities**: Bind all services together into the `EnrichmentService` workflow and expose FastAPI endpoints. 
- **Core LLD Refs**: §6 Workflow, §14 (API Endpoint lists).
- **Output**: The `/api/v1/abeyance/*` REST server.

**Agent 6: Value Ledger Agent**
- **Responsibilities**: Implement mathematical logic to ingest resolved tickets, search topology references, calculate historical deltas, and attribute `attributed_mttr_saving`.
- **Core LLD Refs**: §13 (`baseline_mttr - actual_mttr` logic), Rules 1-6.
- **Output**: `services/value_attribution.py`.

**Agent 7: Validation Agent**
- **Responsibilities**: Build the test harness verifying graph structures properly form, cyclic temporal contexts cluster logically around `midnight`, and ground-truth divergence detection matches expected confidence values.
- **Core LLD Refs**: §15 Visualisations, §16 Phase 7 Testing Constraints.
- **Output**: The Pytest suite and time-travel test runners.

## Step 7 — Execution Strategy (Risk-Minimizing Rollout)

The multi-agent rollout path requires rigorous gating to prevent compounded architectural debt regarding the high-stakes algorithmic dependencies:

1. **Gate 1 - Physical Schema & Query Perf Validate (Phase 1-3 Subset)**: ORM Models, Fast pgvector retrievals, GIN queries. Agents cannot move to temporal math until recursive CTE performance is verified (<50ms).
2. **Gate 2 - The Math Core (Phase 2 & 5)**: Accurate embeddings, time mapping, decay structures. Mathematical unit tests must enforce that Dec 31 wraps seamlessly to Jan 1. Vector space clustering (`可视化 Visualisation 3` in LLD §15) must test positively via isolated PCA verification.
3. **Gate 3 - The Snap Engine Core (Phase 3 & 4)**: The heart of the LLD. Pair matching and multi-graph mapping. 
4. **Gate 4 - Moat Protection & Output (Phase 6)**: Value attribution, sanitized CMDB exporting (LLD §8 Moat requirement).

- **Time-Space Continuum Bounds Checking**: Explicit tests (Pytest fixtures) verifying that `norm_timestamp`, `sin()/cos()`, and `doy` bounds never skew output norms, ensuring the 1536-dim vector never biases time over operational context. (Aligns with Visualization 3, LLD §15).

## Step 10 — Implementation Risk Analysis

1. **Algorithmic Density of Vector Computations (LLD §9 Risk)**:
   - *The Danger*: Executing `target_retrieval` fetches 200 items. Running Cosine Similarity across 200 arrays for every incoming fragment could create CPU lockup.
   - *Mitigation*: Vector mathematics strictly offloaded to the C-powered `pgvector` operators natively within the Postgres DB rather than Python loops (`w_sem × cosine_sim(enriched, enriched)`).
2. **Flywheel Value Nullification (LLD §13 Risk)**:
   - *The Danger*: Tagging exported CMDB entries, but the customer's ITSM fails to record the tag in ticket resolution metadata. The value ledger stays at 0.
   - *Mitigation*: Implement strong reliance on `Rule 4: Licence and CapEx Savings` and `Rule 6: Dark Graph Reduction Index`, ensuring Pedkai calculates deterministic savings rather than relying wholly on external Incident tags.
3. **Recursive CTE Infinite Loops (LLD §10 Risk)**:
   - *The Danger*: Circular dependencies form in `accumulation_edge`.
   - *Mitigation*: Apply `max_hop` execution abort mechanisms inside the `WITH RECURSIVE` queries.
