## 1. System Overview

Pedk.ai is an AI-assisted network operations centre (NOC) platform designed for deployment by licensed UK telecommunications operators. The system ingests real-time telemetry from live network elements — including BSS and OSS data streams — and applies machine learning techniques to detect anomalies, correlate alarm events, and identify degraded network conditions such as sleeping cells that produce no alarms despite active subscriber impact.

The platform generates structured situation reports (SITREPs) and recommends corrective actions to NOC operators. In its initial deployment configuration, all recommendations are advisory only. No network configuration changes are executed without explicit human approval.

The autonomy model is structured across four discrete levels, governed by the `ai_maturity_level` configuration parameter:

| Level | Designation | Behaviour |
|-------|-------------|-----------|
| 0 | Shadow (observe only) | AI processes data; no outputs surfaced to operators |
| 1 | Assisted | AI outputs logged internally; operators work independently |
| 2 | Supervised (Advisory) | AI recommendations presented; human approval required for every action |
| 3 | Autonomous | AI executes pre-approved low-risk actions; human post-hoc review |

**Default deployment level is Level 2 (Supervised/Advisory).** Level 3 is not implemented in the current version and cannot be activated without a separate board-level approval, a new Data Protection Impact Assessment, and a minimum of six months of Level 2 operational data demonstrating a false positive rate below 2%.

## 2. Regulatory Relevance

Pedk.ai Ltd notifies Ofcom of this deployment on a voluntary basis, in accordance with the spirit of General Conditions of Entitlement §A.3 (network reliability obligations) and the Ofcom Statement on AI in Telecoms (2025).

The notification is provided because the system analyses live subscriber-affecting network metrics and generates recommendations that, if followed, may influence the operator's ability to meet its quality of service (QoS) obligations under the Communications Act 2003. Specifically:

- The system processes real-time radio access network KPIs including PRB utilisation, Reference Signal Received Quality (RSRQ), and Channel Quality Indicator (CQI), which directly reflect subscriber-level service quality.
- The system identifies sleeping cells — network elements that have silently degraded — and flags them for operator investigation. Prompt detection of sleeping cells has a direct bearing on the operator's QoS obligations.
- The system computes service impact estimates, including subscriber counts and revenue at risk, to assist operators in prioritising their incident response.

The platform is deployed entirely within the operator's own infrastructure. Pedk.ai Ltd does not access third-party networks and does not hold any network licence. All processing occurs under the operator's existing regulatory authorisation. This notification does not constitute an application for any regulatory approval or exemption.

## 3. Autonomy Model and Human Oversight

Human oversight is the foundational design principle of the Pedk.ai platform at all deployment levels below Level 3.

At Level 2 (Supervised/Advisory), which is the deployment level covered by this notification:

- Every AI-generated recommendation carries a mandatory watermark: *"This content was generated by Pedk.ai AI. It is advisory only and requires human review before action."*
- Three mandatory human gates are enforced before any recommended action can be recorded as executed: (1) SITREP approval, (2) action approval, (3) incident closure.
- No network configuration change of any kind is initiated by the system.
- Emergency service entities — those flagged `is_emergency_service` in the operator's service inventory — are subject to an additional mandatory human gate regardless of the configured autonomy level. This protection cannot be overridden by configuration.

Should the operator elect in future to activate Level 3 (Autonomous), any such action must traverse a seven-gate safety pipeline before execution:

1. **Blast Radius Gate**: Any action affecting more than ten network entities is automatically and irrevocably rejected.
2. **Policy Engine Gate**: The proposed action is evaluated against operator-configured rules. Operators define permitted action types, permitted time windows, and permitted entity classes.
3. **Confidence Gate**: Actions below a configurable confidence threshold (default: 0.75) are automatically escalated to human review.
4. **Emergency Services Gate**: Actions affecting any entity tagged `is_emergency_service` are unconditionally blocked and escalated.
5. **Maintenance Window Gate**: Actions are suppressed during scheduled maintenance windows to prevent conflicts with planned work.
6. **Confirmation Window**: A minimum thirty-second human override window is enforced before any autonomous action proceeds.
7. **Post-Execution Validation Gate**: For five minutes following any autonomous action, the system polls relevant KPIs. If any monitored metric degrades by more than ten per cent relative to the pre-action baseline, the action is automatically rolled back.

An emergency kill-switch endpoint (`POST /api/v1/autonomous/kill-switch`) is available to NOC operators at all times. Invoking the kill-switch marks all recent autonomous actions as rolled back and halts further autonomous execution pending operator review. The kill-switch is authenticated and fully audit-logged.

A complete, tamper-evident audit trail is maintained for all AI recommendations, operator decisions, and automated actions. Every record carries a correlation identifier (`trace_id`) enabling full reconstruction of the decision chain for any incident. Audit records are retained for seven years in accordance with the operator's regulatory obligations under the Communications Act 2003.

## 4. Data Handling

The system processes the following categories of data:

**Network KPI telemetry (non-personal):** Time-series metrics from radio access and core network elements, retained on a thirty-day rolling basis in a TimescaleDB instance hosted within the operator's infrastructure. This data is not personal data in bulk; it represents aggregate performance characteristics of network nodes.

**Alarm and incident data:** Structured alarm events from the operator's existing OSS/BSS systems. Alarm records may carry cell site identifiers that, in combination with other data, could theoretically be linked to location information. Alarm data is retained for thirty days operationally and for seven years as part of incident records for regulatory purposes.

**Customer impact estimates:** Aggregated counts of affected subscribers and estimated revenue at risk. Where these figures are derived from estimated data rather than confirmed billing records, they are flagged `is_estimate: true` throughout the system to prevent false precision in regulatory or financial reporting.

**PII handling:** All prompts transmitted to the external LLM provider (Google Gemini API) are processed by the platform's PII scrubbing service before transmission. The scrubber detects and replaces phone numbers, IMSI, IMEI, email addresses, IP addresses, and account identifiers, and produces a scrub manifest (containing no original values) for audit purposes. The Google Gemini data processing agreement is required to be in place prior to production deployment.

Multi-tenant isolation is enforced at the database query level via mandatory `tenant_id` constraints. No cross-operator data sharing occurs. All data remains within the operator's own infrastructure throughout its lifecycle. For operators requiring data sovereignty, a sovereign mode is available that prevents any data egress from the operator's environment.

## 5. Risk Assessment Summary

The following risks have been identified and mitigated:

- **False positive rate causing operator distraction:** The system is calibrated against a thirty-day shadow mode baseline before any advisory outputs are presented to operators. Target false positive rate is below five per cent before Level 2 is activated. A rate below two per cent is required before Level 3 can be considered. Monthly calibration cycles continue in production. Human oversight at Level 2 means a false positive results in a dismissed recommendation, not a network change.

- **System unavailability affecting network operations:** Because the platform is advisory-only at Level 2, system downtime has no direct impact on network operations. The operator's NOC procedures function independently of Pedk.ai at all times. No operator workflow has a hard dependency on platform availability during the pilot.

- **Data integrity and read-only ingestion:** The platform operates in read-only mode with respect to the operator's existing OSS/BSS data. It does not write to operator systems except via explicitly approved actions at Level 3. The telemetry ingestion pipeline is append-only; no source data is modified.

- **Vendor dependency and platform lock-in:** The platform is built on open-source components (PostgreSQL, TimescaleDB, pgvector, Kafka, FastAPI, Next.js) and is deployed on infrastructure owned and controlled by the operator. No proprietary data format is used for storage. The operator retains full access to all data at all times and is not dependent on Pedk.ai Ltd for continued data access in the event of contract termination.

- **LLM hallucination in generated SITREPs:** AI-generated situation reports are produced using a causal template architecture that constrains the LLM output to structured, verifiable claims derived from the underlying telemetry data. All AI-generated content is watermarked and requires human approval before any associated action is taken. Confidence scores are derived from Decision Memory similarity (cosine distance via pgvector) rather than from the LLM itself, providing an independent quality signal.
